# Exact Evaluation Knowledge

Core concepts for evaluating open-ended AI outputs with unambiguous, reproducible scores.

## Overview

Exact evaluation produces judgment without ambiguity (vs. subjective evaluation like essay grading or AI-as-judge). Two approaches yield exact scores: **functional correctness** (does the output do what's intended?) and **similarity measurements** against reference data (exact match, lexical, semantic).

## Key Concepts

### Functional Correctness

**Definition**: Evaluating a system based on whether it performs the intended functionality.

The ultimate metric for any application — measures whether it actually does what it's supposed to do. Often hard to automate, but works well for tasks with executable/measurable outcomes.

**Key points**:
- Code generation: run generated code against unit tests (a.k.a. *execution accuracy*)
- Game bots: measure score achieved
- Optimization tasks: measure objective (e.g., energy saved)
- Used in benchmarks: HumanEval, MBPP, Spider, BIRD-SQL, WikiSQL

### pass@k Metric

**Definition**: For each problem, generate `k` code samples; problem is "solved" if any sample passes all test cases. Final score = fraction of solved problems.

- pass@1 < pass@3 < pass@10 in expectation (more samples = more chances)
- Standard for code-generation benchmarks

### Reference-Based vs Reference-Free

**Definition**: Reference-based metrics need ground-truth answers; reference-free metrics do not.

- Each reference example: `(input, reference_responses)` — an input may have multiple acceptable responses
- Reference data is generated by humans (gold standard) or increasingly by AI (cheaper, may need review)

### Exact Match

**Definition**: Generated response matches a reference response exactly (binary score).

Works for short, exact responses: math answers, trivia, fill-in-the-blank, account lookups. Variant: accept any output that *contains* the reference (risky — see rules.md).

### Lexical Similarity

**Definition**: How much two texts overlap at the surface (token/character) level.

Two main flavors:
- **Edit distance / fuzzy matching**: count edits (deletion, insertion, substitution; sometimes transposition) to convert text A → text B
- **N-gram overlap**: % of n-grams (unigrams, bigrams, ...) in reference also present in generated text

Common metrics: **BLEU, ROUGE, METEOR, TER, CIDEr**. Output is a sliding scale (0–1).

### Semantic Similarity (Embedding Similarity)

**Definition**: How close two texts are in meaning, computed via embedding vectors.

- Convert each text to an embedding
- Compute distance/similarity (typically **cosine similarity**, range −1 to 1)
- Two identical embeddings → 1; opposite → −1
- Common metrics: **BERTScore**, **MoverScore**

### Embedding

**Definition**: A numerical vector representation that aims to capture the meaning of the original data.

- Vector size typically 100–10,000
- Produced by specialized models: BERT, CLIP, Sentence Transformers, OpenAI/Cohere embeddings APIs
- Quality goal: more-similar inputs produce closer vectors (by cosine similarity)
- Multimodal joint embeddings (CLIP, ULIP, ImageBind) enable cross-modal comparison (e.g., text→image search)

## Cosine Similarity Formula

For embeddings A and B:

```
cosine_sim(A, B) = (A · B) / (||A|| * ||B||)
```

- `A · B` = dot product
- `||A||` = Euclidean (L2) norm of A

## Terminology

| Term | Definition |
|------|------------|
| Execution accuracy | Functional correctness for code (run it, check output) |
| pass@k | Fraction of problems solved with k samples per problem |
| Ground truth / canonical response | Reference correct answer |
| Reference-based | Metric needs reference answers |
| Reference-free | Metric does not need references |
| Edit distance | Min edits to convert one string to another |
| n-gram | Sequence of n consecutive tokens |
| Embedding | Vector capturing semantic content |
| Multimodal embedding | Shared vector space across modalities (text, image, audio) |

## How It Relates To

- **AI-as-a-judge** (next section): subjective alternative when exact methods fail
- **RAG / vector search**: uses embedding similarity for retrieval
- **Data deduplication**: similarity to detect near-duplicates
- **Clustering / anomaly detection**: leverages embedding distances

## Common Misconceptions

- **Myth**: Higher BLEU/ROUGE always means better output.
  **Reality**: On HumanEval, BLEU scores for correct vs. incorrect code were similar — BLEU does not track functional correctness.

- **Myth**: Exact match is "objective" so always safe.
  **Reality**: It marks correct paraphrases as wrong; only fits short, canonical answers.

- **Myth**: Semantic similarity is fully objective.
  **Reality**: Different embedding models produce different vectors; the score is exact *given* embeddings, but choice of embedding model is subjective.

- **Myth**: A correct answer always gets a high reference-based score.
  **Reality**: A novel-but-correct response can score low if the reference set is incomplete (e.g., Adept's Fuyu image captioning case).
