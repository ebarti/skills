# AI Application Pre-Launch Checklist

Use before committing to build, before milestone reviews, and before shipping an AI product to production.

## Before You Start

- [ ] You can state in one sentence why this product exists (existential threat, productivity gain, or strategic exploration)
- [ ] You have decided build vs buy and can justify it
- [ ] You have a defensibility hypothesis and can name the moat (technology, data, or distribution)
- [ ] You know whether the product is critical or complementary, reactive or proactive, dynamic or static

## Use Case and Role of AI

- [ ] Business motivation is documented and reviewed by a non-engineering stakeholder
- [ ] Role of AI vs humans is explicit (AI suggests / AI handles simple cases / AI handles all)
- [ ] Crawl-Walk-Run stage is named for the launch
- [ ] Promotion criteria from one stage to the next are defined and measurable

## Defensibility

- [ ] You can answer: "If a model provider hires three engineers to build this, why do you survive?"
- [ ] If the moat is data, the data flywheel is wired into the product before launch
- [ ] If the moat is distribution, you have an actual distribution channel today
- [ ] You have identified an "incumbent feature" risk and have a counter-positioning plan

## Success Metrics

- [ ] Business metrics are defined (e.g., % messages automated, response time, cost per ticket)
- [ ] Each business metric maps to one or more measurable AI metrics
- [ ] Customer satisfaction tracking is in place, not only volume metrics
- [ ] User feedback collection is wired into the interface

## Usefulness Thresholds

- [ ] Quality threshold is defined and instrumented
- [ ] Latency thresholds are defined for TTFT, TPOT, and total latency
- [ ] Cost per inference threshold is defined
- [ ] Other thresholds (interpretability, fairness, safety) are defined where applicable
- [ ] All thresholds have a measurement plan, not only a number

## Milestone Planning

- [ ] Off-the-shelf model has been evaluated against the goal before scoping work
- [ ] Goals have been re-validated after that evaluation
- [ ] Plan accounts for the last-mile challenge (reserve 4x time for the final 20%)
- [ ] Demo velocity is not being used to forecast launch dates

## Stack Layering

- [ ] You have started at the application development layer (prompts, context, evaluation, interface)
- [ ] You have descended to model development only when the app layer was insufficient
- [ ] You have descended to infrastructure only when serving, monitoring, or compute forced you to
- [ ] You have used the lightest adaptation technique that meets the threshold (prompt < context < finetune < pre-train)

## Maintenance Readiness

- [ ] You have a versioning plan for prompts, models, and datasets
- [ ] You have an evaluation harness that runs on every model swap
- [ ] You have a documented model-swap exercise (vendor change, price drop, deprecation)
- [ ] You have monitored regulatory exposure (GDPR, IP, export controls, compute restrictions)
- [ ] You have a contingency plan for losing access to your current GPU vendor or model provider

## Discipline Fit

- [ ] You have classified the work: AI engineering, ML engineering, or full-stack engineering (or a blend)
- [ ] The team has the skills the classification requires
- [ ] If the work is AI engineering, the team owns prompts, context, evaluation, and interface
- [ ] If the work is ML engineering, the team owns architecture, training, datasets, and inference

## Red Flags

Stop and address if you find:

- The product can be replicated by an incumbent in two weeks with three engineers and no special data
- A killer demo with no defined usefulness threshold
- Launch date set from demo velocity, ignoring the last mile
- A single model provider with no fallback and no evaluation harness
- Metrics that only count volume (messages handled) and not satisfaction
- AI is critical to the product but quality bar is set as if it were complementary
- Proactive feature shipped without a higher quality bar
- Open-ended outputs being evaluated like close-ended classification
- IP-sensitive content generated by a model whose training data provenance is unknown

## Quick Reference

| Aspect | Ideal | Acceptable | Red Flag |
|--------|-------|------------|----------|
| Why we build | Existential or clear productivity case | R&D learning project | "Because AI" |
| Role of AI | Classified on all 3 axes | Classified on critical/complementary | Unclassified |
| Defensibility | Data flywheel + distribution | Data flywheel only | Pure prompt wrapper |
| Threshold | Quality + latency + cost + other | Quality + latency | Vibes only |
| Milestone plan | 4x last-mile reserve | 2x last-mile reserve | Linear from demo |
| Stack entry | App layer first | App + selective model layer | Custom infra day one |
| Adaptation | Prompt engineering | Prompt + context + RAG | Pre-training from scratch |
| Maintenance | Versioned, swappable, monitored | Versioned only | None |
