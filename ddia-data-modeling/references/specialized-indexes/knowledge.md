# Specialized Indexes Knowledge

Core concepts for indexes that go beyond single-attribute B-trees and LSM-trees: multidimensional, full-text, and vector indexes.

## Overview

Standard B-tree and LSM-tree indexes support range queries on a single attribute (or a fixed prefix via concatenated indexes). Specialized indexes exist to query multiple dimensions simultaneously (geo, color, time+temperature), to search by keywords inside text documents, and to retrieve documents by semantic similarity using vector embeddings.

## Key Concepts

### Concatenated Index

**Definition**: A multicolumn index that combines several fields into one key by appending columns in a fixed order (e.g., `(lastname, firstname)`).

**Key points**:
- Useful for prefix lookups in declared order (`lastname`, or `lastname + firstname`)
- Useless for querying a non-prefix column alone (e.g., `firstname` only)

### Multidimensional Index

**Definition**: An index that allows querying several columns simultaneously rather than one column at a time.

**Key points**:
- Required when queries constrain ranges on two or more dimensions at once
- A concatenated index cannot answer such queries efficiently

### R-tree / Bkd-tree

**Definition**: Specialized spatial index structures that divide space so nearby points cluster in the same subtree.

**Key points**:
- Used by PostGIS (via PostgreSQL's GIST facility) for geospatial data
- Alternative: translate 2D location to a single number via a space-filling curve, then use a regular B-tree

### Geospatial Query

**Definition**: A query that finds records within a geographic region (e.g., all restaurants inside a map's rectangular viewport).

**Key points**:
- Requires simultaneous range constraints on latitude AND longitude
- Naturally a 2D problem; not solvable efficiently by single-dimension indexes

### Full-Text Search

**Definition**: Searching a collection of text documents by keywords that may appear anywhere in the text.

**Key points**:
- Conceptually a multidimensional query where each possible term is a dimension (1 if present, 0 if absent)
- Often involves language-specific tokenization, stemming, synonyms, and typo handling

### Term

**Definition**: A word (or token) extracted from a document during indexing; one dimension in the full-text vector space.

### Inverted Index

**Definition**: A key-value structure mapping each term to the list of document IDs that contain that term.

**Key points**:
- The list of IDs per term is called a *postings list*
- Postings can be stored as sparse bitmaps (bit `n` set if doc `n` contains the term) for efficient bitwise AND across multiple terms

### Postings List

**Definition**: The list of document IDs associated with a given term in an inverted index.

### N-gram Index

**Definition**: An inverted index built over all length-`n` substrings (e.g., trigrams of `hello` are `hel`, `ell`, `llo`) instead of full words.

**Key points**:
- Enables arbitrary substring search and even regular expression queries
- Indexes are large compared to word-based ones

### Edit Distance / Levenshtein Automaton

**Definition**: A measure of how many single-character insertions/deletions/substitutions separate two strings; a Levenshtein automaton enables efficient lookup of all terms within a given edit distance.

**Key points**:
- Used by Lucene to handle typos in queries or documents
- Built atop a finite-state automaton (trie-like) over indexed terms

### Vector Embedding

**Definition**: A vector of floating-point values produced by an embedding model that represents a document as a point in a high-dimensional space.

**Key points**:
- Often >1,000 dimensions
- Semantically similar inputs produce vectors that are close together
- Generated by neural-network embedding models (Word2Vec, BERT, GPT, multimodal models)

### Semantic Search

**Definition**: Search that matches by meaning rather than exact keywords, by comparing vector embeddings of query and documents.

**Key points**:
- Distance measured by cosine similarity or Euclidean distance
- Core retrieval mechanism for retrieval-augmented generation (RAG)

### Vector Index

**Definition**: An index over a collection of vector embeddings that returns documents whose vectors are closest to a query vector.

**Key points**:
- R-trees do not work well at high dimensionality
- Specialized algorithms: Flat, IVF, HNSW

### Approximate Nearest Neighbor (ANN)

**Definition**: An algorithm that returns vectors *close to* the query vector, trading some accuracy for large speed/memory wins versus exhaustive search.

### IVF (Inverted File) Index

**Definition**: A vector index that clusters the vector space into partitions (centroids) and only compares the query against vectors in selected partitions (probes).

**Key points**:
- More probes -> more accurate, slower
- Can miss neighbors that fall in a different partition

### HNSW (Hierarchical Navigable Small World) Index

**Definition**: A vector index built as a multi-layer graph; each layer's nodes are vectors and edges connect nearby vectors. Queries descend from a sparse top layer to a dense bottom layer.

**Key points**:
- Approximate but high-quality results
- Widely implemented (Faiss, pgvector, most vector DBs)

## Terminology

| Term | Definition |
|------|------------|
| Concatenated index | Multi-column index keyed by appended column values |
| Multidimensional index | Index supporting simultaneous range queries on multiple columns |
| R-tree / Bkd-tree | Spatial index grouping nearby points in shared subtrees |
| Space-filling curve | 1D mapping of 2D coordinates usable in a B-tree |
| Term | A word/token used as a dimension in full-text indexing |
| Inverted index | term -> postings list mapping |
| Postings list | List of document IDs containing a term |
| N-gram | Length-`n` substring used as an index key |
| Edit distance | Minimum single-character edits to transform one string into another |
| Vector embedding | Floating-point vector representing document semantics |
| Cosine similarity | Cosine of the angle between two vectors |
| ANN | Approximate nearest neighbor search |
| IVF | Partition-based approximate vector index |
| HNSW | Multi-layer graph-based approximate vector index |

## How It Relates To

- **B-trees / LSM-trees**: These are the single-dimension foundation; specialized indexes extend or replace them when queries don't fit one ordered key.
- **Vectorized query processing**: Different "vector"; vectorized processing means batched bits in CPU pipelines, not embedding vectors.
- **AI / RAG**: Vector indexes are the retrieval substrate that feeds context into LLMs.

## Common Misconceptions

- **Myth**: A concatenated index on `(lat, lon)` answers map-area queries efficiently.
  **Reality**: It can only filter by latitude range OR longitude range, not both simultaneously.

- **Myth**: "Vector" in semantic search is the same as "vectorized" query execution.
  **Reality**: Embedding vectors are arrays of floats describing locations in semantic space; vectorized execution refers to batched bit-level processing.

- **Myth**: Vector indexes give exact results.
  **Reality**: IVF and HNSW are approximate; only flat indexes are exact (and slow).
