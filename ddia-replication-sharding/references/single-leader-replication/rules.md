# Single-Leader Replication Rules

Design and operational guidance for building and operating leader-based replicated systems.

## Core Rules

### 1. Choose Sync vs Async Based on Durability Requirements

Synchronous and asynchronous each have hard tradeoffs.

- **Sync**: Follower guaranteed up-to-date; safe for failover. But any sync-follower outage halts all writes.
- **Async**: Writes never blocked by follower availability. But confirmed writes can be lost if the leader fails before replicating.
- **Never make all followers synchronous** — a single node outage halts the cluster.
- **Use semisynchronous** (one sync + rest async) for a practical balance: durable on at least 2 nodes, but tolerant of single follower failures.
- **Use majority quorum** (e.g., 3 of 5 sync) for systems built on consensus protocols.

### 2. Prefer Logical (Row-Based) Replication for Operability

Replication log type determines whether you can do zero-downtime upgrades.

- **Logical/row-based**: Decoupled from storage engine; supports cross-version replication. Required for zero-downtime upgrades. Easy to consume externally (CDC).
- **WAL shipping**: Highest performance and simplest, but locks you to identical software versions on leader and followers. Upgrades require downtime.
- **Avoid statement-based**: Breaks on `NOW()`, `RAND()`, autoincrement, `WHERE` ordering, and stored procedures with side effects. Use only when you can guarantee determinism (e.g., VoltDB).

### 3. Implement Read-Your-Writes Consistency in the Application

The database alone often doesn't guarantee a user sees their own writes.

Pick the technique that fits your edit pattern:

- **Edit-rare data** (e.g., user's own profile): Always read user's own data from the leader; everyone else's from followers.
- **Edit-frequent data**: Track timestamp of last write; read from leader for N seconds (e.g., 1 minute) after a write. Or monitor follower lag and skip followers that are behind.
- **Strong guarantee**: Client remembers a logical timestamp (LSN) from its last write; replica must be caught up to that timestamp before serving the read. Block or reroute if not.
- **Cross-device**: Centralize the timestamp metadata (can't keep it per-device). Route all of a user's devices to the same region if reading from the leader.

### 4. Use Sticky Sessions for Monotonic Reads

To prevent users seeing time go backward:

- Route a given user's reads to the **same replica** consistently.
- Use a **hash of user ID** to pick the replica (deterministic, no shared state).
- On replica failure, accept temporary loss of monotonicity or fall back to leader.

### 5. Pick the Most Up-to-Date Follower for Failover

Failover is dangerous; minimize data loss.

- **Sync/semisync**: Promote the follower the leader was waiting for — guaranteed current.
- **Async**: Pick the follower with the **highest LSN** (or equivalent log position).
- **Never promote a stale follower** — losing seconds is tolerable; losing days is catastrophic.
- **Implement fencing**: Prevent old leader from accepting writes if it returns (avoid split brain).

### 6. Tune Failover Timeouts Carefully

The timeout to declare a leader dead is a tradeoff.

- **Too long**: Extended outage during real failures.
- **Too short**: Spurious failovers from load spikes or network glitches make things worse.
- Plan for **30 seconds** as a typical starting point; tune based on observed behavior.
- Prefer **safe handoff** for planned maintenance instead of relying on timeout-based detection.

### 7. Handle Cross-System Coordination Carefully on Failover

Discarded writes after failover can corrupt other systems.

- The GitHub MySQL incident: a stale promoted leader reused autoincrement IDs that Redis still keyed against → wrong users saw private data.
- **Don't share IDs** generated by the database with external systems unless you're sure failover discards are safe.
- Consider UUIDs or globally-unique IDs for keys shared across systems.

## Guidelines

- **Use async replication** for read-scaling — sync to many followers is unreliable.
- **Set up new followers without downtime**: snapshot + log catch-up; use tools like WAL-G, Litestream, Percona XtraBackup.
- **Archive replication logs** to object storage for backup/disaster recovery and follower bootstrap.
- **Default to NewSQL** (strongly consistent distributed DB with ACID) when you can — simpler than working around eventual consistency in app code.
- **Prefer manual failover** if your team is small and downtime is acceptable; automatic failover has many failure modes.
- **Don't pretend async is sync** — handle the staleness explicitly or you'll hit it in production.

## Exceptions

- **Read-only workloads with stale-tolerant data** (analytics dashboards, search indexes): Eventual consistency is fine; no need for monotonic reads.
- **Geo-distributed apps**: Reading from leader from far regions adds latency; accept some staleness instead.
- **High write throughput with many followers**: Fully async is the only practical option.
- **VoltDB-style deterministic systems**: Statement-based replication is safe and very compact.

## Quick Reference

| Decision | Recommendation |
|----------|---------------|
| Write durability critical | Sync to at least one follower (semisync) |
| Read scaling priority | Async, many followers |
| Zero-downtime upgrades | Logical/row-based replication |
| Best raw performance | WAL shipping (with version lockstep) |
| User sees own writes | Read from leader after own write, or use LSN timestamp |
| User sees consistent timeline | Sticky session by user ID hash |
| Picking new leader | Highest log position; sync follower if available |
| Avoid split brain | Fencing + careful timeout tuning |
| External system shares DB IDs | Use UUIDs, not autoincrement |
